Section 01

Purpose and Scope

M3 defined how the OCP should work. This document validates that it actually works against real operator data during the founding cohort pilot. The distinction matters. A specification describes intended behavior; a validation framework confirms that intended behavior is observed in production conditions with live operators, live telemetry payloads, and live routing decisions.

Three things are validated: that the OCP correctly ingests telemetry from each live operator, that the CRS calculation pipeline produces scores that match independently verified calculations, and that routing weight assignments comply with policy bounds and concentration limits. These three layers can fail independently. An operator whose telemetry ingestion is clean may still produce a CRS calculation mismatch if the OCP's component weighting is misconfigured. An operator whose CRS is correct may still receive a non-compliant routing weight if policy bounds are not enforced correctly. The validation framework therefore runs all three layers in sequence and requires each to pass independently.

This document covers the live validation phase only. Pre-pilot onboarding checks are covered in M4(1) Cohort Onboarding and Technical Readiness Audit. Post-pilot ongoing monitoring is outside this document's scope.

Design Decision — Sequential vs Parallel Validation Layers Running validation layers in sequence rather than in parallel means a telemetry ingestion failure blocks CRS validation, and a CRS failure blocks routing weight validation. This creates a dependency chain that could slow validation for operators with early-layer failures. The alternative — running all three layers simultaneously — would generate misleading results: a CRS calculated from malformed telemetry would fail for the wrong reason. Sequential execution ensures each failure is diagnosed at the correct layer, which produces more actionable remediation guidance. The cost is validation time. The benefit is diagnostic precision.
Section 02

Assumption Registry and Open Questions

ASSUMPTION-01 Confirmed

CRS component weights are fixed at the values established in M3: Verified Inference Score 40%, uptime rate 30%, p95 latency 20%, telemetry freshness 10%. Validation uses these weights for independent recalculation. Any deviation in the OCP's applied weights from these values constitutes a CRS calculation failure.

ASSUMPTION-02 Confirmed

Independent CRS recalculation is performed by Superstructure operations using the raw telemetry payload data logged by the OCP ingestion system. The recalculation tool is a deterministic script applying the published formula — not a manual calculation. This ensures the independent calculation is repeatable and auditable.

ASSUMPTION-03 Confirmed

The OCP recalculates CRS on a 60-second cycle. Validation checks are run against a minimum of five consecutive scoring cycles (5 minutes of live data) to confirm scoring stability, not just a single calculation pass.

ASSUMPTION-04 Open

The OCP exposes a query endpoint that allows Superstructure operations to retrieve the raw input values used for any given CRS calculation, not just the final score. Without this, independent recalculation must rely on the telemetry payload log rather than the OCP's actual inputs — which would not detect cases where the OCP read the correct payload but applied the wrong field mapping internally. Confirmation of this endpoint's availability is pending from Post Fiat core.

UNKNOWN-01 — Routing weight recalculation interval during pilot

M3 defines routing weight as derived from CRS on each 60-second scoring cycle. Whether the pilot uses the same interval or a longer recalculation period for stability during the initial live phase is not confirmed. This validation spec uses the 60-second interval as the baseline.

Section 03

Three-Layer Validation Model

Every operator in the founding cohort pilot must clear all three validation layers. The layers run in sequence. Layer 1 must pass before Layer 2 begins. Layer 2 must pass before Layer 3 begins.

Layer 1 — Telemetry Ingestion OCP receiving correct data from operator

Validates that the OCP is receiving telemetry payloads from this operator at the correct interval, that payloads pass schema validation, that signatures are valid, and that no delivery gaps exceed the consecutive miss threshold. A failure at this layer means the OCP cannot produce a trustworthy CRS for this operator — routing decisions based on stale or malformed data are unreliable.

Layer 2 — CRS Calculation OCP scoring pipeline producing correct output

Validates that the OCP's CRS output matches an independently computed score derived from the same raw telemetry inputs. Passes only if the delta between OCP score and independent score is within 2 points across five consecutive scoring cycles. A failure at this layer means the OCP's scoring pipeline has a configuration error — wrong weights, wrong field mapping, or stale input data — that would produce incorrect routing decisions even if ingestion is clean.

Layer 3 — Routing Weight Assignment OCP routing logic compliant with policy and concentration limits

Validates that the routing weight assigned to this operator falls within the policy bounds specified in the institution's allocation policy, that the weight responds correctly to CRS changes, and that no concentration limit violations are produced. A failure at this layer means the OCP's weight assignment logic is not enforcing the constraints that protect institutional allocations from operator concentration risk.

Validation Complete

All Three Layers Cleared — Operator Enters VALIDATED State

Middleware fires OPERATOR_TELEMETRY_VALIDATION_COMPLETE. Operator is confirmed as a live, scored participant in the founding cohort. Institutional routing decisions for this operator are trusted as OCP-verified.

Step 02 — Layer 2

CRS Calculation Mapping and Validation

Layer 2 validates that the OCP's CRS calculation pipeline produces correct output. A correctly operating OCP will produce a CRS that matches an independently computed score derived from the same raw telemetry inputs within the defined tolerance. A mismatch indicates a pipeline error that must be diagnosed and resolved before live routing decisions can be trusted.

CRS Formula — Published Component Weights

CRS = (VI_score × 0.40) + (uptime_rate × 0.30) + (latency_score × 0.20) + (freshness_score × 0.10)
All component inputs normalized to 0.00–1.00 scale before weighting. Final CRS expressed as 0–100 integer (multiply result by 100).

Component Mapping — Telemetry Field to CRS Input

40%
Verified Inference Score
Source field: verifiedInferenceScore

Used directly as normalized input (already 0.00–1.00). Highest weight component — reflects operator's contribution quality to the network beyond uptime. A new operator with no scored periods defaults to the VI score floor defined by Post Fiat core (see UNKNOWN-01 in M4(1)).

30%
Uptime Rate
Source field: uptimeRate24h

Used directly as normalized input. Reflects 24-hour rolling uptime. Validation checks that the OCP uses the rolling 24-hour value, not an instantaneous point-in-time reading, which would produce artificially volatile CRS scores.

20%
Latency Score
Source field: latencyP95Ms (transformed)

Derived from p95 latency via inversion: latency_score = max(0, (400 - latencyP95Ms) / 400). A p95 latency of 0ms produces a latency score of 1.00; a p95 latency of 400ms produces 0.00; values above 400ms are floored at 0.00. Validation confirms the OCP applies this transformation, not a raw latency comparison.

10%
Telemetry Freshness
Source field: timestamp (derived)

Derived from payload age at time of CRS calculation: freshness_score = max(0, (30 - age_seconds) / 30). A payload received at exactly 30 seconds old produces 0.00; a payload received instantly produces 1.00. Validation confirms the OCP calculates freshness at scoring time, not at ingestion time.

Layer 2 Validation Procedure

  • 1

    Retrieve Raw Telemetry Inputs

    For each of five consecutive 60-second scoring cycles, retrieve the raw field values that the OCP used as inputs: verifiedInferenceScore, uptimeRate24h, latencyP95Ms, and payload timestamp. If ASSUMPTION-04 is confirmed (OCP exposes a query endpoint), retrieve directly from OCP. Otherwise, use the OCP ingestion log as the source of record.

    Proceed: inputs retrieved for all five cycles
    Fail: ERR_VALIDATION_INPUT_UNAVAILABLE — cannot retrieve OCP inputs; escalate to Superstructure operations review
  • 2

    Run Independent CRS Calculation

    Apply the published formula to each set of retrieved inputs using the Superstructure operations recalculation script. The script applies component transformations (latency inversion, freshness derivation) identically to the OCP's documented implementation. Produces an expected CRS value for each cycle.

    Proceed: independent scores calculated for all five cycles
  • 3

    Compare OCP Output vs Independent Calculation

    For each cycle, compute the delta between OCP CRS output and independent calculation. A delta of 0–2 points is within tolerance (rounding differences in floating-point arithmetic). A delta above 2 points on any single cycle triggers ERR_CRS_CALCULATION_MISMATCH. A consistent delta of the same magnitude across all five cycles suggests a systematic weight misconfiguration rather than a transient error.

    Pass: delta ≤ 2 points on all five cycles
    Fail: ERR_CRS_CALCULATION_MISMATCH — delta > 2 on any cycle
  • 4

    Diagnose Mismatch Type (on failure only)

    Determine which component is producing the mismatch by recalculating each component independently. A weight mismatch (OCP applying wrong percentage weights) will show consistent proportional error across all operators. A field mapping error (OCP reading the wrong telemetry field for a component) will show component-specific error that varies by operator. A stale input error (OCP using payload from a prior cycle) will show temporal drift in the mismatch.

    Mismatch type determines remediation path — see Step 5

Step 03 — Layer 3

Routing Weight Adjustment Mechanism

Layer 3 validates that the OCP correctly translates CRS scores into routing weights — that weights fall within policy bounds, respond appropriately to CRS changes, and do not create concentration violations. A correctly operating OCP will produce routing weights that are proportional to CRS within the policy floor and ceiling, and will adjust those weights within the defined response window when CRS changes.

Layer 3 Validation Procedure

Layer 3 runs three specific checks against live routing weight data:

Check R-01 — Policy Bounds Compliance Weight within floor and ceiling for operator state

For each routing weight assignment logged during the 4-hour observation window, verify the assigned weight is within the applicable floor and ceiling for this operator's current state. An operator with CRS above 70 must receive a weight between 10% and 60%. An operator with CRS below 70 must be capped at 30%. An operator with maintenanceDeclared: true may receive 0%.

Pass: All weight assignments within policy bounds for operator state.

Fail: ERR_ROUTING_WEIGHT_POLICY_VIOLATION — any assignment outside applicable bounds.

Check R-02 — CRS Responsiveness Weight adjusts within 60 seconds of CRS change

During the observation window, identify any CRS change of 5 or more points in a single scoring cycle for any pilot operator. Confirm that the operator's routing weight is updated within the following 60-second cycle to reflect the new CRS. A weight that does not respond to a material CRS change within one cycle indicates the OCP's weight recalculation trigger is not functioning.

Pass: Weight updated within 60 seconds of any CRS change ≥ 5 points.

Fail: ERR_ROUTING_WEIGHT_STALE — weight not updated within one cycle of material CRS change.

Check R-03 — Concentration Limit Enforcement No cohort-wide concentration violations produced

Verify that no routing weight assignment across the full pilot cohort produces a concentration limit violation: no more than 2 operators from the same jurisdiction or infrastructure provider exceeding 40% routing weight simultaneously. Also verify weights sum to 100% across all active operators per institution.

Pass: No concentration violations detected; weights sum to 100%.

Fail: ERR_CONCENTRATION_LIMIT_BREACH or ERR_WEIGHT_NORMALIZATION_FAILURE.

Note on Layer 3 Failures

Layer 3 failures are OCP configuration failures, not operator failures. An operator cannot fix a routing weight policy violation — the OCP's weight assignment logic must be corrected. Layer 3 failures therefore trigger an OCP configuration review rather than an operator remediation workflow. The affected operator's VALIDATED state is withheld until the OCP issue is resolved and Layer 3 is re-run.

Step 05

Remediation Process for Failing Operators

Remediation paths differ by failure layer because the root causes differ. Layer 1 failures are operator infrastructure failures. Layer 2 failures are OCP configuration failures. Layer 3 failures are OCP policy enforcement failures. An operator should not be required to take remediation action for a Layer 2 or Layer 3 failure — those require Superstructure operations or Post Fiat core to resolve the OCP configuration.

Layer 1 Failure — Operator Telemetry Infrastructure 48-hour remediation window

Operator is responsible for resolving Layer 1 failures. Superstructure operations notifies the operator within 2 hours of failure detection with the specific check ID, failure detail, and error code. The operator must:

  • Identify the root cause within 4 hours of notification
  • Submit a written remediation plan within 12 hours
  • Resolve and request retest within 48 hours of the original failure

Retest runs a fresh 4-hour observation window. A passing retest clears Layer 1 and Layer 2 validation begins. A second consecutive Layer 1 failure after the 48-hour window triggers escalation per the M4(1) remediation framework.

Layer 2 Failure — OCP CRS Calculation Configuration 24-hour resolution window

Superstructure operations owns the Layer 2 remediation. The operator is notified that validation is paused pending OCP investigation — no action required from the operator. Superstructure operations must:

  • Identify the mismatch type (weight misconfiguration, field mapping error, or stale input) using the Layer 2 diagnostic procedure within 4 hours
  • Escalate to Post Fiat core if the error is in the OCP's core calculation logic rather than a configuration setting within 8 hours
  • Resolve and re-run Layer 2 validation within 24 hours

A consistent mismatch pattern across multiple operators indicates a systemic OCP configuration error. In that case, all affected operators' validations are paused simultaneously and a single OCP fix resolves all pending Layer 2 checks.

Layer 3 Failure — OCP Routing Weight Policy Enforcement 24-hour resolution window

Superstructure operations owns the Layer 3 remediation. Operator is notified that validation is paused pending OCP investigation. Remediation steps:

  • Identify the specific policy enforcement failure (bounds violation, stale weight, concentration breach, or normalization failure) within 4 hours
  • Correct the OCP's weight assignment logic or policy configuration within 24 hours
  • Re-run Layer 3 validation with a fresh 4-hour observation window after the fix is deployed

Concentration limit breaches affecting multiple operators are resolved by adjusting the OCP's cohort-wide weight allocation algorithm, not by penalizing any individual operator.

Section 10

Error Codes and Typed Failure Reasons

ERR_TELEMETRY_NOT_RECEIVED
Layer 1 — L1-01 Fail

Zero telemetry payloads received from this operator during observation window. Operator infrastructure is either offline or misconfigured. Immediate block.

ERR_TELEMETRY_DELIVERY_RATE_LOW
Layer 1 — L1-02 Fail

Delivery rate below 98% during 4-hour window. Actual delivery rate and poll count logged for operator diagnosis.

ERR_TELEMETRY_GAP
Layer 1 — L1-03 Fail

Four or more consecutive missed polls detected. Gap start and end timestamps logged. Indicates periodic outage pattern rather than steady-state delivery degradation.

ERR_SCHEMA_VALIDATION_FAILURE
Layer 1 — L1-04 Fail

One or more payloads failed schema validation. Specific missing or malformed fields logged per payload for operator diagnosis.

ERR_SIGNATURE_INVALID
Layer 1 — L1-05 Fail

One or more payloads had missing or invalid Ed25519 telemetrySignature. Indicates signing infrastructure misconfiguration.

ERR_TIMESTAMP_DRIFT
Layer 1 — L1-06 Fail

Payload timestamp more than ±5 seconds from OCP receipt time. Indicates clock synchronization issue on operator infrastructure. Affects freshness score accuracy.

ERR_VALIDATION_INPUT_UNAVAILABLE
Layer 2 — L2-01 Fail

OCP CRS inputs not retrievable for one or more of the five required scoring cycles. Blocks independent recalculation. Escalates to Superstructure operations review.

ERR_CRS_CALCULATION_MISMATCH
Layer 2 — L2-02 Fail

Delta between OCP CRS output and independent calculation exceeds 2 points on any cycle. Mismatch type (weight, field mapping, stale input) logged from diagnostic step. OCP configuration failure — not operator failure.

ERR_ROUTING_WEIGHT_POLICY_VIOLATION
Layer 3 — L3-01 Fail

Routing weight assignment outside applicable floor/ceiling for operator's current state. Specific violation (above ceiling, below floor, degraded cap exceeded) logged. OCP configuration failure.

ERR_ROUTING_WEIGHT_STALE
Layer 3 — L3-02 Fail

Routing weight not updated within 60 seconds of a CRS change of 5 or more points. OCP weight recalculation trigger not functioning. OCP configuration failure.

ERR_CONCENTRATION_LIMIT_BREACH
Layer 3 — L3-03 Fail

More than 2 operators from same jurisdiction or infrastructure provider exceeding 40% routing weight simultaneously. OCP cohort-wide allocation algorithm failure.

ERR_WEIGHT_NORMALIZATION_FAILURE
Layer 3 — L3-03 Fail

Routing weights across active operators do not sum to 100% for one or more institutions. OCP normalization step not executing correctly after weight updates.
